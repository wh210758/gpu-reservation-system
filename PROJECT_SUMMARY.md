# 实验室 GPU/服务器算力预约与看板系统 - 项目总结报告

## 1. 项目概况

**项目名称**：实验室 GPU/服务器算力预约与看板系统 (GPU Reservation & Dashboard System)
**项目目标**：为高校/研究机构的实验室提供一款轻量级、高透明度的算力预约与监控看板系统，解决 GPU 资源抢占、排期不透明、沟通成本高等核心痛点。
**交付标准**：在 MVP (最小可行性产品) 阶段达成资源状态可视化、排班规范化、组件极简部署，并具备强悍的异常处理与并发冲突防护能力。

## 2. 系统架构与技术栈

本项目采用**纯前后端分离与 Agent 探针解耦**的轻量化架构，所有模块均坚持“低依赖、高可用”的设计原则。

*   **前端 / 看板端 (Frontend)**：
    *   **技术栈**：Vue 3 (Composition API) + Vite + TypeScript + Tailwind CSS
    *   **核心组件**：基于 `@fullcalendar/vue3` 的拖拽式排期表、基于 `vue-echarts` (ECharts 5) 的实时算力与显存时序波动图。
    *   **通信方式**：通过 Axios 采用 HTTP 短轮询与后端接口保持状态同步。
*   **后端 / API 层 (Backend)**：
    *   **技术栈**：Python 3 + FastAPI + SQLite3 + Pydantic
    *   **核心能力**：提供高性能的 RESTful API，并内置 SQLite 本地数据库，处理时序与空间层面的重叠预约拦截。
*   **探针端 (Agent)**：
    *   **技术栈**：纯 Python 3 (无厚重框架) + `psutil`
    *   **核心能力**：使用原生的 `nvidia-smi` 进程抓取 GPU 核心利用率与显存情况，并结合 `psutil` 侦测宿主机的 CPU/RAM 数据。借助内置 `http.server` 提供支持“阅后即焚(ACK)”机制的 API 供内网或后端拉取数据。

## 3. 核心功能及业务逻辑实现

### 3.1 可视化资源排期与防冲突机制
*   **前端双重防线**：在 FullCalendar 配置中开启了 `selectOverlap` 与 `eventOverlap` 的拦截，阻止用户在 UI 层面发起与他人重叠的预约。添加了请求遮罩层 (Loading Overlay) 防止网络波动时的连击重发。
*   **后端时序仲裁**：所有新建预约请求 (`POST /api/reservations`) 均受严格校验：
    1.  **畸形请求拦截**：预约结束时间必须晚于开始时间，且禁止预定早于当前时间 (含 5 分钟容差) 的历史时段。
    2.  **空间重叠仲裁**：数据库级别运用 `start_time < [已有结束时间] AND end_time > [已有开始时间]` 逻辑进行强一致性排他校验，一旦冲突即返回 `409 Conflict`。

### 3.2 高可用监控探针与环境免疫
*   **Agent 异常静默与重试**：探针运行于 `while True` 守护线程中，在硬件指令调用超时、SQLite 缓冲池高负载或网络断开时，通过 `try-except` 捕获异常，仅输出本地日志并等待下一个周期重试，保障探针进程的绝对存活。
*   **硬件解耦免疫**：应对宿主机显卡驱动缺失 (`nvidia-smi` 崩溃或未找到) 的情况，Agent 将退化为仅采集 CPU 与 RAM 信息的模式 (作为一个特殊的虚拟节点 `SYS` 抛出)，完美实现“在 GPU 字段优雅返回空列表”的系统健壮性目标。

### 3.3 响应式自适应大屏看板
*   项目针对 PC 端宽屏 (1920x1080) 与小屏笔记本 (1280x720) 均做了布局响应式适配。
*   `App.vue` 调整了网格视窗的溢出控制行为，配合 CSS Grid 响应式断点（`xl:grid-cols-12`），在小屏幕设备上自动折叠为单列上下滚动模式，避免了内容挤压与横向滚动条的出现。
*   集成了智能诊断标签机制，根据资源利用率 (>5%) 与当前预约时间块进行交叉对比，能够直观高亮出“违规占用”、“空闲中”或“合规运行”等业务状态。

## 4. 后续演进路线 (Next Steps)

1.  **鉴权与认证模块**：目前用户身份采用前端传参伪造（`当前用户`），下一步将引入 JWT Token 与基于角色的访问控制 (RBAC) 系统，区分 Admin 与 Student 权限。
2.  **通知告警机制**：对被判定为“违规占用”的状态，接入企业微信 / 钉钉机器人或邮件通知，实现自动化管理闭环。
3.  **时序数据库迁移**：随着探针轮询产生海量的指标日志，未来可考虑将 SQLite 升级为 InfluxDB 或 Prometheus 等专用时序数据库，进一步提升图表的历史数据拉取性能。
