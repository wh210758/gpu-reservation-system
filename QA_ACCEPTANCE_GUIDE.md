# 项目负责人 (PM / QA) 全链路验收与联调指南

作为本项目的负责人（D同学），你的核心职责是：把控进度、合并代码 (Merge PR)、并在本地将前端、后端、探针三大模块同时跑起来进行全链路验证 (E2E Testing)。

本文档将手把手教你如何在自己电脑上配置并启动“全栈环境”，以及针对各模块的具体验收测试步骤。

---

## 一、 负责人专属：全栈联调环境搭建

你需要同时开启 3 个终端窗口，分别运行后端、前端和探针。

### 1. 前置准备与拉取代码

建议每天验收前，先确保拉取了 dev 分支的最新代码：

```bash
cd gpu-reservation-system
git checkout dev
git pull origin dev
```

### 2. 终端 1：启动 Backend (后端引擎)

打开第一个终端，进入 backend 目录，启动 API 服务器。

```bash
cd backend
# 激活你之前创建的虚拟环境
source venv/bin/activate  # Windows: venv\Scripts\activate
# 安装最新依赖（如果有同学更新了包）
pip install -r requirements.txt
# 启动 FastAPI
uvicorn app.main:app --reload
```

成功标志：控制台输出 Application startup complete.，此时你的后端监听在 http://127.0.0.1:8000。

### 3. 终端 2：启动 Frontend (前端看板)

打开第二个终端，进入 frontend 目录，启动网页。

```bash
cd frontend
# 安装最新依赖
npm install
# 启动 Vite 开发服务器
npm run dev
```

成功标志：控制台输出 Local: http://localhost:5173/。此时打开浏览器即可看到前端界面。

### 4. 终端 3：启动 Agent (模拟探针)

由于你本机可能没有 N 卡（NVIDIA GPU），你需要要求探针工程师 (C同学) 提供一个 Mock（模拟）模式，或者你自己手动修改一下探针脚本，让它发送假数据。
打开第三个终端，进入 agent 目录。

```bash
cd agent
source venv/bin/activate  # Windows: venv\Scripts\activate
# 运行探针（假设 C 同学写好了 main.py）
python main.py
```

成功标志：控制台每 10 秒打印一次 数据上报成功：{"cpu": 15%, "gpu_util": 85%...}。

## 二、 严格验收步骤（Definition of Done 测试）

当你把三个终端都跑起来后，请按照以下步骤对三位同学的产出进行刁钻的测试验收：

### 验收任务 A：后端 API 防冲突与健壮性测试 (B同学)

作为测试，你完全不需要写代码，直接使用 FastAPI 自带的 Swagger UI。

1. 打开测试台：浏览器访问 http://127.0.0.1:8000/docs。
2. 测试用例 1 (正常预约)：
   - 找到 POST /api/reservations 接口。
   - 点击 Try it out，输入一段未来的预约时间（例如 10:00 - 12:00）。
   - 期待结果：返回 HTTP 状态码 200 或 201。
3. 测试用例 2 (故意制造冲突 - 致命测试)：
   - 再次请求同一个接口，输入时间 11:00 - 13:00（时间段有重叠！）。
   - 期待结果：后端必须无情拦截！返回 HTTP 状态码 409 Conflict，并附带错误信息（例如：“该时段与张三的预约冲突”）。
   - 注意：如果后端返回了 200，或者抛出了 500 报错，请直接将这段操作截图发回给 B同学，打回重做。
4. 测试用例 3 (时光倒流测试)：
   - 尝试预约昨天的时间。期待结果：返回 400 Bad Request（不能预约过去）。

### 验收任务 B：前端 UI 体验与防抖测试 (A同学)

1. 测试用例 1 (多分辨率挤压测试)：
   - 浏览器打开 http://localhost:5173。
   - 按 F12 打开开发者工具，点击左上角“设备模拟切换 (Device Toggle)”。
   - 将分辨率设为 iPad (768x1024) 和 iPhone (375x667)。
   - 期待结果：日历表格不应该出现惨不忍睹的文字重叠或严重溢出，图表依然能看清。
2. 测试用例 2 (帕金森连击防抖测试 - 致命测试)：
   - 在前端点击“提交预约”按钮时，疯狂快速连击鼠标 10 次。
   - 期待结果：前端应该弹出 Loading 遮罩层屏蔽你的点击，并在数据库中只生成一条预约记录。
   - 注意：如果数据库生成了 10 条重复记录，说明前端没做防抖/状态锁，直接打回重做。
3. 测试用例 3 (冲突直观展示)：
   - 确认刚才在 Swagger 里成功的 10:00-12:00 的预约，在前端日历中显示为灰色/红色不可用状态。

### 验收任务 C：探针极端环境容错测试 (C同学)

1. 测试用例 1 (后端崩溃模拟 - 致命测试)：
   - 确保终端 3 (Agent) 正在运行。
   - 故意切断终端 1 (Backend)（按 Ctrl + C 关闭 FastAPI 服务器）。
   - 观察终端 3：Agent 绝不应该抛出一大段红色 Traceback 报错然后闪退！
   - 期待结果：Agent 应该安静地打印一条警告：无法连接到后端服务器，10秒后重试...。
   - 重新启动终端 1 (Backend)，观察 Agent 是否能自动恢复上报。
2. 测试用例 2 (环境丢失模拟)：
   - 随便找一台没有装显卡驱动的电脑跑 agent/main.py。
   - 期待结果：脚本不能崩溃，应该能正常将系统 CPU 和内存发送给后端，只是返回的 GPU 列表为空 []。

## 三、 终极全链路模拟跑通 (大作业演示脚本)

在你们的小组答辩前一天，请你作为负责人亲自走一遍以下流程（这也是给老师演示的最佳剧本）：

1. 老师视角/大屏展示视角：打开监控看板，目前 GPU 占用率为 0%。
2. 学生A视角：在自己电脑网页上，预约 明天 14:00 - 18:00 的算力。
3. 学生B视角：在网页上尝试预约 明天 15:00 的算力，系统弹窗提示“已被学生A占用，预约失败”，学生B表示叹服。
4. 后台视角：你悄悄开启探针模拟程序，发送了一段 GPU 95% 的高负载假数据。
5. 结果展示：前端看板在大屏上自动跳动，出现了资源暴涨的折线图，全场鼓掌。

给大模型的提示：如果在验收过程中遇到任何 Bug，请将你的操作步骤和观察到的实际结果发给大模型。例如：“我是 QA，前端在执行多次点击时发起了多个请求。请给我一段可以在 Vue3 中拦截重复提交的 hook 代码，或者指导前端 A同学 如何使用按钮的 disabled 状态。”
