## 1. 日常开发高频命令

### 1.1 获取与更新代码

每天开始写代码的第一件事，永远是拉取最新代码！

- `git clone <仓库地址>` —— 第一次参与项目时，下载全部代码。
- `git pull origin dev` —— 每天必点：将远端 `dev` 分支的最新代码拉取到本地当前分支。
- `git fetch -a` —— 刷新本地能看到的远端分支列表（如果有队友建了新分支你看不到，敲这个）。

### 1.2 分支操作 (Branch)

永远不要在 main 和 dev 分支上直接写代码！

- `git branch` —— 查看本地所有分支（带 `*` 的是你当前所在的分支）。
- `git checkout <分支名>` —— 切换到已有分支（例如 `git checkout dev`）。
- `git checkout -b <新分支名>` —— 最常用：基于当前代码，创建并立刻切换到一个新分支（例如 `git checkout -b feat/login`）。
- `git branch -d <分支名>` —— 删除本地的某个分支（只能删别人，不能删自己当前所在的分支）。

### 1.3 提交代码 (Commit)

当你写完一个功能，准备存档时：

- `git status` —— 随时敲：查看当前哪些文件被修改了，哪些还没暂存（红色代表未暂存，绿色代表已暂存）。
- `git add .` —— 将当前目录下所有修改过的文件放入暂存区（准备提交）。
- `git commit -m "feat(前端): 新增了日历组件"` —— 正式提交存档。引号里一定要写清楚改了什么！
- `git push origin <你的分支名>` —— 将本地的存档推送到云端仓库（例如 `git push origin feat/login`）。

## 2. 后悔药专区

写错代码了？提交错文件了？别慌，Git 全都能救回来。

### 2.1 "我刚改了文件，但还没 git add，想全部恢复原状"

- `git restore <文件名>` —— 丢弃某个文件的修改，恢复到上次提交的状态。
- `git restore .` —— 慎用：丢弃当前所有未保存的修改！彻底清空你的瞎折腾。

### 2.2 "我已经 git add 了，但我后悔了不想提交这个文件"

- `git restore --staged <文件名>` —— 把文件从暂存区踢回未暂存区，你的代码改动不会丢失。

### 2.3 "我已经 git commit 了，但发现注释写错了，或者漏了一个文件"

- `git commit --amend -m "新的注释"` —— 修改最近一次的提交注释。如果配合 `git add`，还可以把漏掉的文件塞进上一次的提交里，不会产生新的提交记录。

### 2.4 "我提交了好几次，发现思路全错了，想回退到昨天某个版本"

- `git log` —— 查看历史提交记录，找到你想回退到的那次提交的 commit ID (一长串乱码，如 a1b2c3d)。按 q 退出查看。
- `git reset --hard <commit ID>` —— 极度危险：让代码时光倒流到那一次提交，这之后的修改将彻底灰飞烟灭！
- `git reset --soft <commit ID>` —— 安全倒流：撤销那之后的提交记录，但你写的代码还会保留在你的编辑器里（变成未暂存状态）。

## 3. 冲突与暂存

### 3.1 "我要切换分支，但系统不让，提示我有未提交的代码" (Stash)

有时候你代码写了一半（不想 commit 制造垃圾记录），但队长叫你紧急去 `dev` 分支测个 bug：

- `git stash` —— 神技：把当前所有没提交的代码“塞进口袋”藏起来。此时你的工作区会变得干干净净，可以随便切分支了。
- `git stash pop` —— 测完 bug 切回自己的分支后，敲这个命令，把刚才“口袋”里的代码拿出来继续写。

### 3.2 "如何优雅地合并代码并解决冲突"

假设你在 `feat/ui` 分支，队友的代码已经合入了 `dev`：

1. 先切换到 dev 并更新：`git checkout dev` -> `git pull origin dev`
2. 切回你的分支：`git checkout feat/ui`
3. 将 dev 的新代码合并到你的分支：`git merge dev`
   - 如果没有冲突：Git 自动弹出一个合并信息，输入 `:wq` 即可（如果是 Vim 编辑器）。
   - 如果产生冲突 (Conflict)：命令行会提示 Merge conflict in xxx。
     - 此时打开 VSCode，找到冲突文件，你会看到 `<<<<<<< HEAD` 和 `=======` 之类的奇怪符号。
     - VSCode 上方通常会有“Accept Current Change (保留你的)”、“Accept Incoming Change (保留队友的)”、“Accept Both (都要)”的快捷按钮。
     - 点选满意的代码，保存文件。
     - 最后再次执行 `git add .` 和 `git commit -m "fix: 解决合并冲突"` 即可！

## 4. 咱们组的黄金规范流

请在心里默念这套操作，它能帮你省去 99% 的麻烦：

1. 上班/开工：
   
   ```bash
   git checkout dev
   git pull origin dev
   git checkout -b feat/我的新功能   # 永远基于最新的 dev 建分支
   ```

2. 写代码，测试跑通后：
   
   ```bash
   git add .
   git commit -m "feat(模块名): 我干了啥"  # 提交信息参考下方的规范
   git push origin feat/我的新功能
   ```

3. 下班/交接：在 GitHub / GitLab / Gitee 网页端，向 `dev` 分支发起 Pull Request (PR)。严禁自己强行 merge！让项目负责人检查后再点合并按钮。

### 附录：Commit Message 规范前缀

- feat: 新增了一个功能 (Feature)
- fix: 修复了一个 Bug
- docs: 只改了文档 (如 README.md)
- style: 改了代码格式 (空格、缩进等，不影响运行逻辑)
- refactor: 重构了代码 (不是新功能，也不是修 Bug，比如优化了某个函数的写法)
- test: 增加或修改了测试用例
- chore: 构建过程或辅助工具的变动 (如改了包配置文件)
